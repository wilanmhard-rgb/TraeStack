# Office Hours

Turn a rough idea into a buildable brief before implementation starts.

## When To Use

Use this skill when the user:
- wants to brainstorm a new product or feature
- asks whether an idea is worth building
- wants help narrowing scope before coding
- needs a design brief, problem statement, or execution wedge

## Goals

Produce a short artifact that answers:
- who the user is
- what painful problem exists right now
- what the narrowest useful wedge is
- what alternatives were considered
- what should be built first
- what success and failure look like

## Workflow

1. Clarify the idea in one sentence.
2. Identify the target user and the current workaround.
3. Pressure-test urgency, frequency, and differentiation.
4. Narrow to the smallest credible first version.
5. List 2-3 alternatives and why they were rejected.
6. End with a concrete recommendation.

## Output

Create a concise brief in markdown when useful. Keep it practical, not visionary fluff.

Suggested sections:
- Problem
- User
- Current workaround
- Wedge
- Alternatives
- Risks
- Recommended next step

## Operating Notes

- Ask focused questions when the idea is ambiguous.
- Prefer narrowing scope over expanding it.
- Tie every recommendation to user pain or execution cost.
- If the request is already a concrete implementation task, hand off to planning or engineering instead of repeating ideation.
